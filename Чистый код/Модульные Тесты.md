### Три закона TDD
- Первый закон: не пишите код продукта, пока не напишите отказной модульный тест
- Второй закон: не пишите модульный тест в объёме больше, чем необходимо для отказа (невозможность компиляции является отказом)
- Третий закон: не пишите код продукта в объёме больше, чем необходимо для прохождения текущего отказного теста

### О чистоте тестов
- Тесты на скорую руку равносильны полному отсутствию тестов, если не хуже
- Тесты должны изменяться по мере развития кода продукта
- Чем примитивнее тест, тем труднее его изменить
- Тестовый код не менее важен, чем код продукта

### Тесты как средство обеспечения изменений
- Именно модульные тесты обеспечивают гибкость, удобство сопровождения и возможность повторного использования кода
- Наличие автоматизированного пакета модульных тестов, охватывающих код продукта, имеет важнейшее значение для чистоты и ясности архитектуры
- Тесты обеспечивают возможность внесения изменений

### Чистые тесты
- Чистый тест характеризуется удобочитаемостью, ясностью, простотой и выразительностью

### Предметно-ориентированный язык тестирования
- Стройте тесты на предметном языке — тесты должны оперировать понятиями домена, а не низкоуровневыми API
- Скрывайте детали реализации — используйте вспомогательные функции и служебные программы, чтобы убрать ненужные технические подробности
- Тестовый API развивается итеративно — не проектируйте заранее, улучшайте по мере переработки тестового кода
- Тесты должны быть лаконичными и читаемыми — ясно показывать, что проверяется, без лишнего шума
- Тесты формируют свой язык — набор вспомогательных функций создаёт удобный и понятный «язык тестирования»

### Двойной стандарт
- Тесты должны быть простыми, лаконичными и выразительными, но от них не требуется такая же эффективность, как от кода, который работает в реальной среде
- Тестовый код может подчиняться упрощённым стандартам — в тестах важнее читаемость, выразительность и лаконичность, а не максимальная эффективность
- Скрывайте ненужные детали — функции лучше спрятать за понятными методами, чтобы тест фокусировался на поведении, а не на механике
- Чтение теста должно быть плавным — структура проверок должна позволять взгляду двигаться в одном направлении
- Используйте абстракции для компактного выражения состояния — это сокращает визуальный шум и ускоряет понимание результата
- При тестировании допустимы компромиссы по эффективности — в тестовой среде можно жертвовать производительностью ради ясности и удобства чтения кода
- Фокус на намерении, а не на ресурсах — в тестах важен смысл проверки, а не оптимизация памяти или процессора

### Одна проверка на тест
- Каждый тест проверяет одно условие — тест должен быть сфокусированным и легко читаемым
- Разделяйте подготовку, выполнение и проверку — структура «дано–когда–тогда» улучшает читаемость
- Разделяйте проверки при необходимости — если проверяется несколько аспектов, создавайте отдельные тесты
- Сокращайте дублирование через вспомогательные функции — общие шаги выносите в отдельные методы или базовые структуры
- Баланс между чистотой и практичностью — допустимо несколько проверок в одном тесте, если это упрощает понимание

### Одна концепция на тест
- В каждой тестовой функции должна тестироваться одна концепция
- Количество проверок на концепцию должно быть минимальным

### F.I.R.S.T.
- Чистые тесты должны обладать пятью характеристиками, название которых образует сокращение F.I.R.S.T.
- **Быстрота (Fast)** — тесты должны выполняться быстро
- **Независимость (Independent)** — тесты не должны зависеть друг от друга
- **Повторяемость (Repeatable)** — тесты всегда должны давать повторяемые результаты в любой среде
- **Очевидность (Self-validating)** — результатом выполнения теста должен быть логический признак: либо True, либо False
- **Своевременность (Timely)** — тесты должны создаваться своевременно, модульные тесты пишутся непосредственно перед кодом продукта
